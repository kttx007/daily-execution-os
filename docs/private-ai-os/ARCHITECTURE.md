# Private AI OS Architecture

> 状态：Bootstrap Blueprint
> 目标仓库：建议新建私有仓库 `kttx007/private-ai-os`
> 当前文件仅存放非敏感架构，正式业务数据不得在公开仓库落盘。

## 1. 设计目标

建立一套可被 ChatGPT、Codex、Claude Code、IDE Agent 和人工团队共同读取的长期业务知识系统。

系统必须解决：

- 跨对话上下文丢失；
- 产品参数串型号与 AI 幻觉；
- 客户、项目、产品、SOP、素材散落；
- 外部资料收藏后无法复用；
- 每日总结只形成流水账，不能转化为业务资产；
- 不同 AI 工具接手时重复解释背景。

## 2. 总体原则

1. **聊天记录不是事实库。**聊天内容只能作为待整理输入。
2. **产品事实必须有证据。**无来源内容标记 `UNKNOWN`，不得按行业常识补齐。
3. **同一事实只保留一个权威位置。**其他文件引用，不复制多份。
4. **知识按需加载。**Agent 只读取完成当前任务所需的最小权威上下文。
5. **业务敏感信息只进私有仓库。**客户、成本、底价、订单、合同、售后争议不得放公开仓库。
6. **大文件不直接堆入 Git。**PDF、图片、视频原件优先放云盘/对象存储，GitHub 保存索引、摘要、版本、哈希和链接。

### 定位

- `10_projects/`：例如“法国客户 K6 第二批订单”“阿里首页 8 屏升级”“本地单据系统 V1”。
- `20_areas/`：长期持续维护的权威知识与运营责任。
- `30_assets/`：可直接复用或发布的规格书文案、销售话术、SOP、模板、培训材料。
- `40_resources/`：外部文章、竞品网页、行业资料、原始参考。
- `50_skills/`：如何完成某类任务，而不是某次任务的结果。

## 5. 五层知识模型

```text
Layer 0  Agent Contract
  安全边界、事实规则、读取顺序、审批权限

Layer 1  Current State
  当前项目状态、阻塞、下一工作包、验证状态

Layer 2  Canonical Domain Knowledge
  产品事实、客户卡、售后规则、渠道规则、业务口径

Layer 3  Evidence & Resources
  PDF、图片、证书、聊天截图、订单、网页来源及索引

Layer 4  Archive
  旧版本、被否决方案、过期资料、历史日志
```

## 6. 知识状态枚举

所有关键条目必须明确标记：

- `FACT`：有可核验来源的事实；
- `DECISION`：经过负责人确认的业务决策；
- `INFERENCE`：基于事实的推断，不能对外当事实；
- `ASSUMPTION`：为推进工作暂时采用的假设；
- `UNKNOWN`：缺失、冲突或未核实；
- `RETIRED`：已废止的旧口径。

## 7. 每日对话与知识库关系

```text
ChatGPT / Codex / Claude 对话
        ↓
00_inbox/conversations/YYYY-MM-DD.md
        ↓ 每日整理
60_reviews/daily/YYYY-MM-DD.md
        ↓ 每周审核与知识晋升
10_projects / 20_areas / 30_assets / 50_skills
        ↓
60_reviews/weekly/YYYY-Www.md
```

每日摘要只记录：发生了什么、做了什么、待办是什么、哪些内容可能值得沉淀。

只有经过以下检查后，内容才能进入正式知识区：

1. 是否有来源或责任人确认；
2. 是否长期有效；
3. 是否已有权威条目，避免重复；
4. 是否包含敏感信息；
5. 应进入项目、责任区、资产、资源还是技能；
6. 是否需要标记版本与适用范围。

## 8. 大文件策略

| 类型 | GitHub | 云盘/对象存储 |
|---|---|---|
| Markdown、YAML、JSON、代码 | 主存储 | 可备份 |
| 小型稳定 PDF | 可存，但需版本控制 | 同步原件 |
| 证书扫描件、高清图片 | 保存索引与低清预览 | 保存原件 |
| 视频、工程素材、大 ZIP | 只保存索引、摘要、链接、哈希 | 主存储 |
| 客户合同、PI、付款凭证 | 仅私有且严格授权；优先保存索引 | 加密云盘/业务系统 |

## 9. 自动化边界

允许自动化：

- 生成每日/周报草稿；
- 建立待审核知识卡；
- 更新索引、标签、文件链接；
- 检测冲突、过期字段、缺失证据；
- 生成 Codex/Claude 的最小上下文包。

必须人工确认：

- 产品参数与认证对外口径；
- 客户报价、成本、底价、免费支持边界；
- 合同、付款、售后责任承诺；
- 删除权威事实或覆盖历史版本；
- 将内部内容转为公开内容。

## 10. 落地顺序

1. 新建 `private-ai-os` 私有仓库；
2. 写入控制文件与目录骨架；
3. 迁移每日/周报与索引；
4. 建立产品知识 Schema 和首批产品事实卡；
5. 建立证据索引，不急于搬所有大文件；
6. 建立客户项目与 Skill；
7. 跑 7 天人工审核流程；
8. 验证后再增加自动化。
